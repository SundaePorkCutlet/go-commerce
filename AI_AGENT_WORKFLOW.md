# AI Agent Workflow (Production-minded)

이 문서는 AI 에이전트에게 작업을 맡길 때 품질을 유지하기 위한 실행 기준이다.
목표는 "빠른 생성"이 아니라 "검증 가능한 결과"다.

## 1) 작업 요청 템플릿

아래 형식으로 요청한다.

```md
[Task]
- 한 줄 목표:
- 배경/문제:
- 성공 기준(측정 가능):

[Scope]
- 수정 가능 파일:
- 수정 금지 파일:
- 비기능 요구사항(성능/보안/가독성):

[Output]
- 변경 파일 목록
- 핵심 변경 요약(왜 필요한지)
- 리스크/트레이드오프
- 검증 명령 3개 이상
- 롤백 방법
```

## 2) 에이전트 출력 필수 항목

에이전트는 반드시 아래를 포함해야 한다.

- 변경 파일 목록 (파일 경로 정확히)
- 변경 이유 (무엇이 아니라 왜)
- 실패 가능성/회귀 리스크
- 검증 절차 (재현 가능한 명령)
- 미확인 사항/가정

## 3) 검증 체크리스트 (머지 전)

- [ ] 코드 경로/심볼이 실제로 존재한다 (`rg` 기반 확인)
- [ ] 핵심 API/플로우 1개 이상 실제 실행 확인
- [ ] 문서와 코드가 서로 모순되지 않는다
- [ ] 오류/예외 경로 1개 이상 점검했다
- [ ] 과장 표현 없이 현재 구현 범위와 한계를 명확히 적었다

## 4) 멱등/재시도 관련 기준

- 동일 논리 작업에는 동일 idempotency key를 사용한다
- 재시도는 backoff + jitter를 적용한다
- 멱등성과 jitter는 대체 관계가 아니다
  - 멱등/상태: 중복 적용 방지
  - jitter: 재시도 동시 폭주 완화

## 5) 관측성 기준

- 가능하면 메트릭/로그/트레이스 중 최소 1개 이상 검증 근거를 남긴다
- "작동함" 대신 "어떤 신호로 확인했는지"를 기록한다
  - 예: HTTP status, DB row 변화, trace span 증가

## 6) PR/면접 설명용 포맷

아래 4줄 요약으로 설명 가능해야 한다.

1. 문제 정의: 무엇이 막혔는가
2. 접근 방식: AI에 무엇을 맡기고 무엇을 직접 검증했는가
3. 결과: 기능/품질/속도 어떤 변화가 있었는가
4. 한계: 아직 남은 리스크와 다음 액션

## 7) 금지 사항

- 검증 없이 "완료"라고 판단하지 않는다
- 에러 로그를 무시하고 문서만 수정하지 않는다
- 구현되지 않은 내용을 구현된 것처럼 쓰지 않는다
