# AI Agent Rules (Practical)

실무 작업 시 AI 에이전트에게 적용하는 최소 규칙.

## 1. 작업 시작 규칙

- 목표/완료조건을 1~2줄로 먼저 정리한다.
- 수정 범위(파일/디렉토리)와 제외 범위를 명시한다.
- 분석만 하지 말고, 가능한 범위까지 구현+검증까지 진행한다.

## 2. 변경 규칙

- 요청과 직접 관련된 최소 범위만 수정한다.
- 기존 구조/네이밍/스타일을 따른다.
- 모르는 상태에서 추측으로 변경하지 않는다.

## 3. 검증 규칙

- 완료 전 최소 3가지 이상 검증한다.
  - 코드 경로/심볼 확인
  - 실행 또는 API 호출 결과 확인
  - 실패/예외 케이스 1개 이상 확인
  - 문서/설명과 구현 일치 여부 확인
- 검증 없이 "완료"라고 말하지 않는다.

## 4. 보고 규칙

작업 완료 시 아래 4가지를 고정으로 보고한다.

1. 변경 파일 목록
2. 변경 이유(왜 필요한지)
3. 검증 방법과 결과
4. 남은 리스크/후속 작업

## 5. 안전 규칙

- destructive 명령(강제 리셋/강제 삭제)은 사용하지 않는다.
- 사용자가 만든 기존 변경사항은 되돌리지 않는다.
- 구현되지 않은 내용을 구현된 것처럼 말하지 않는다.

## 6. 요청 템플릿

```md
이 문서(AI_AGENT_RULES_HONGJUNHO.md) 기준으로 진행해줘.

[목표]
- (한 줄)

[범위]
- 수정 파일/디렉토리:
- 제외:

[완료조건]
- (측정 가능하게)

[보고 형식]
1) 변경 파일
2) 변경 이유
3) 검증 결과
4) 리스크/후속 작업
```

## 7. Project-specific conventions (go-commerce)

### 7-1. 서비스 구조/레이어

- 서비스는 `USERFC`, `PRODUCTFC`, `ORDERFC`, `PAYMENTFC` 단위로 분리되어 있다.
- 기본 레이어 흐름은 `handler -> usecase -> service -> repository`를 따른다.
- 비즈니스 로직은 handler에 넣지 않고 usecase/service에 둔다.

### 7-2. API/모델 네이밍

- 공개 HTTP 경로는 기존 패턴(`/api/v1/...`, 일부 `/v1/...`)을 유지한다.
- Request/Response JSON 키는 기존 모델 태그와 일치해야 한다.
- 모델 필드/JSON 키를 추측으로 변경하지 않는다.
- 이벤트 구조체는 기존 `...Event` 네이밍 규칙을 유지한다.

### 7-3. 문서/스펙 동기화

- Handler의 요청/응답 스키마 변경 시 Swagger 주석도 함께 업데이트한다.
- Swagger 결과물(`docs.go`, `swagger.json`, `swagger.yaml`)은 코드와 불일치가 없어야 한다.
- 면접/운영 문서의 설명은 실제 코드 상태(구현/미구현)와 일치해야 한다.

### 7-4. 관측성/운영 관련 변경

- 트레이싱 관련 변경 시 context 전파가 끊기지 않는지 우선 확인한다.
- 메트릭/로그/트레이스 설명은 "현재 구현 범위"와 "한계"를 분리해서 작성한다.
- Kafka/멱등성/재시도 관련 변경은 중복 처리 방지와 실패 시나리오를 함께 점검한다.

### 7-5. 프론트엔드(대시보드) 변경

- 프론트의 API 예시는 백엔드 실제 스키마(`items` vs `products` 등)와 맞춰 유지한다.
- 데모/테스트용 바디는 검증 가능한 값으로 구성한다(예: 가격 불일치로 실패하지 않게).
- UI 문구가 아키텍처를 과장하지 않도록 현재 구현 상태 기준으로 작성한다.
